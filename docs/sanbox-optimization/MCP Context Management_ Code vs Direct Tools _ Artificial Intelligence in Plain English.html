Is MCP Failing? Here’s What I Learned About Context Management

Model Context Protocol looked perfect on paper. Then I checked my token usage and its bad…

Gaurav Shrivastav
7 min read
·
12 hours ago

I was excited about MCP. Finally, a standardized way to connect AI agents to tools and data sources. No more custom integrations for every single pairing. Just implement MCP once and unlock an entire ecosystem.

Then I actually tried using it.

Non members can read here for free

I connected two MCP servers , just two , and checked my context usage. 20,000 tokens. That’s roughly 10% of my available context window, and I hadn’t even sent a message yet. Not a single tool had been called. This was just loading the tool definitions.

Something was clearly wrong.

The Problem Nobody Talks About

Turns out, MCP has a dirty secret that becomes obvious once you scale beyond toy examples: context management is a disaster.

When you connect MCP servers the traditional way, two things happen that quietly destroy your agent’s efficiency:

1. Tool definitions bloat your context upfront

Every single tool definition gets loaded directly into your model’s context window. A typical definition might look like this:

gdrive.getDocument
    Description: Retrieves a document from Google Drive
    Parameters:
        documentId (required, string): The ID of the document
        fields (optional, string): Specific fields to return
    Returns: Document object with title, body, metadata, permissions, etc.

Now multiply that by dozens or hundreds of tools. You’re burning through thousands of tokens before your agent does anything useful.

2. Intermediate results consume even more tokens

Here’s where it gets worse. When tools execute, their results flow back through the model’s context. Every. Single. Time.

Let’s say you ask your agent: “Download my meeting transcript from Google Drive and attach it to the Salesforce lead.”

The traditional MCP flow looks like this:

TOOL CALL: gdrive.getDocument(documentId: "abc123")
→ Returns full transcript text (loaded into context)

TOOL CALL: salesforce.updateRecord(
    objectType: "Lead",
    recordId: "00Q5f000001abcXYZ",
    data: { "Notes": [ENTIRE TRANSCRIPT PASTED HERE] }
)

Notice what happened? That transcript flowed through the model twice. For a 2-hour meeting, that’s easily 50,000 additional tokens. And if your document is large enough, you might hit context limits entirely.

In traditional MCP, every tool definition and result passes through the model’s context

The Real-World Breaking Point

When Lutra AI connected to Linear and Intercom’s official MCP servers, they discovered both returned massive JSON blobs with no predefined schemas. Real-world data isn’t clean. It’s messy, nested, and huge.

The problem compounds when you need to:

  • Filter a 10,000-row spreadsheet for specific entries
  • Process multiple API responses sequentially
  • Chain together several tool calls with intermediate data

Your context fills up fast. Your response times slow down. Your costs skyrocket.

Why This Matters More Than You Think

I kept wondering, why didn’t anyone catch this earlier? Then it hit me: most demos use toy examples.

Fetching a single document? Fine. Looking up one database record? No problem. But production workflows involve real data at scale. Multiple sources, large payloads, complex transformations.

That’s when MCP’s architecture starts breaking down.

The Solution: Stop Treating Tools Like Function Calls

Here’s the insight that changes everything: LLMs are excellent at writing code.

Instead of having models directly call MCP tools, what if we presented those tools as code APIs? Let the agent write code to interact with them.

Anthropic published a detailed post about this approach. Cloudflare independently arrived at the same conclusion and called it “Code Mode”. The core idea is identical.

Here’s how it works:

1. Present tools as a file system

Instead of loading all tool definitions upfront, structure them like this:

servers/
├── google-drive/
│   ├── getDocument.ts
│   ├── ... (other tools)
│   └── index.ts
├── salesforce/
│   ├── updateRecord.ts
│   └── index.ts

Each tool becomes a file the agent can read on-demand.

2. Let agents discover tools progressively

The agent explores the filesystem to find what it needs:

  • List ./servers/ to see available integrations
  • Read specific tool files only when needed
  • Load definitions just-in-time rather than all upfront

This approach drops token usage from 150,000 to 2,000 , a 98.7% reduction.

3. Write code instead of chaining tool calls

Our Google Drive to Salesforce example becomes:

import * as gdrive from './servers/google-drive';
import * as salesforce from './servers/salesforce';
const transcript = (await gdrive.getDocument({  
    documentId: 'abc123'  
})).content;

await salesforce.updateRecord({
    objectType: 'Lead',
    recordId: '00Q5f000001abcXYZ',
    data: { Notes: transcript }
});

Notice what’s different? The transcript stays in the execution environment. It never flows through the model’s context. The agent just sees the code it wrote and any explicit logs or return values.

With code execution, intermediate data stays in the sandbox , only final results reach the model

Why Code Execution Changes Everything

This isn’t just about saving tokens. The benefits cascade:

Progressive Disclosure

Load only what you need, when you need it. Search for “salesforce” tools and read just those definitions. The rest stay on disk.

Context-Efficient Results

Filter and transform data before it reaches the model:

const allRows = await gdrive.getSheet({ sheetId: 'abc123' });
const pending = allRows.filter(row => row.Status === 'pending');
console.log(`Found ${pending.length} pending orders`);
console.log(pending.slice(0, 5)); // Only show first 5

The agent sees 5 rows instead of 10,000.

Powerful Control Flow

Loops, conditionals, error handling , all with normal code patterns:

let found = false;
while (!found) {
    const messages = await slack.getChannelHistory({  
        channel: 'C123456'  
    });
    found = messages.some(m =>  
        m.text.includes('deployment complete')
    );
    if (!found) await sleep(5000);
}

Try implementing that with individual MCP tool calls. You’d be alternating between the model and sleep commands, burning context with each iteration.

Privacy Preservation

Sensitive data can flow through your workflow without ever entering the model’s context. An agent harness can even tokenize PII automatically:

// What the agent sees:
[
  { email: '[EMAIL_1]', phone: '[PHONE_1]' },
  { email: '[EMAIL_2]', phone: '[PHONE_2]' }
]
// Real data flows from source to destination
// but never through the model

State Persistence

Agents can save their work and build reusable skills:

// Save intermediate results
await fs.writeFile('./workspace/leads.csv', csvData);

// Later execution picks up where it left off
const saved = await fs.readFile('./workspace/leads.csv');

Over time, your agent builds a library of proven functions it can reuse.

But Wait, Does This Mean MCP Is Useless?

No, and this is the subtle part that’s easy to miss.

MCP still provides something valuable: a uniform interface. Even if you’re not using direct tool calls, MCP gives you:

  • Standardized connection patterns
  • Automatic API discovery
  • Consistent authorization handling
  • Documentation that agents can parse

As Cloudflare put it: “MCP tools are really just an RPC interface with attached documentation. We don’t have to present them as tools.”

You’re essentially using MCP as a standardized API layer, then letting code agents interact with those APIs efficiently.

The Academic Backing

This isn’t just industry intuition. The paper “Executable Code Actions Elicit Better LLM Agents” demonstrated that agents perform better when they can write code to orchestrate actions.

The research showed:

  • Better task completion rates
  • More robust error handling
  • Improved ability to handle complex multi-step workflows
  • Reduced hallucination in tool usage

Code gives models a structured way to think through problems while staying grounded in executable logic.

Implementation Reality Check

I should mention: this approach adds complexity. You need:

  • A secure code execution environment
  • Proper sandboxing and resource limits
  • Monitoring and safety controls
  • State management for long-running sessions

These aren’t trivial. Direct tool calls avoid this infrastructure overhead.

But the tradeoff is clear: pay the complexity cost once at the infrastructure level, then scale to hundreds or thousands of tools without context rot.

Where Does This Leave Us?

The MCP community has built something impressive , thousands of servers, widespread adoption, SDKs for every major language. But we’ve been using it wrong.

Traditional approach:

  • Load all tool definitions → context bloat
  • Pass all results through model → token waste
  • Chain individual calls → latency and cost

Code execution approach:

  • Discover tools on-demand → minimal context
  • Process data in sandbox → efficient workflows
  • Orchestrate with code → powerful composition

The protocol is solid. The usage pattern needs to evolve.

I’ve already shifted my agent architectures toward code execution. The difference is night and day , faster responses, lower costs, better handling of complex workflows.

What’s Your Experience?

Have you run into context limits with MCP? Are you using code execution, or sticking with traditional tool calls? And here’s the controversial bit: do we even need specialized agent protocols, or should we just give LLMs secure code execution environments and let them figure it out?
